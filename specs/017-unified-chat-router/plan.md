# Implementation Plan: Unified Chat Router

**Branch**: `017-unified-chat-router` | **Date**: 2026-04-09 | **Spec**: [spec.md](spec.md)

## Summary

Replace four separate HTTP endpoints with a single `POST /v1/chat` entry point. A GPT-4o-mini intent router classifies each message (confidence-gated, with clarification fallback) and dispatches to the appropriate existing service — `ExtractionService`, `ConsultService`, `RecallService`, or `ChatAssistantService`. Completed consult results are persisted to a new `consult_logs` table. All four old routes are removed; the feedback route is preserved.

---

## Technical Context

**Language/Version**: Python 3.11  
**Primary Dependencies**: FastAPI 0.115, Pydantic 2.10, SQLAlchemy async, Alembic, OpenAI SDK, Langfuse  
**Storage**: PostgreSQL (pgvector) — new `consult_logs` table via Alembic migration  
**Testing**: pytest (`asyncio_mode = "auto"`), ruff, mypy --strict  
**Target Platform**: Linux server (Railway)  
**Project Type**: Web service (FastAPI)  
**Performance Goals**: Intent classification adds ≤150ms overhead (Groq Llama 3.1 8B ~100ms p50)  
**Constraints**: mypy --strict must pass; no raw dicts crossing module boundaries; all LLM calls traced via Langfuse  
**Scale/Scope**: Single new endpoint replacing 4; one new DB table; one new config role

### Step 3 — Schemas: ChatRequest and ChatResponse
**File**: `src/totoro_ai/api/schemas/chat.py`

```python
from totoro_ai.api.schemas.consult import Location   # reuse existing typed model

class ChatRequest(BaseModel):
    user_id: str
    message: str
    location: Location | None = None   # NOT dict — typed Pydantic model

class ChatResponse(BaseModel):
    type: str
    message: str
    data: dict[str, Any] | None        # None for clarification / assistant
```

**Do NOT use `dict | None`** for location — the spec is wrong here. `Location` is the correct type (Constitution Section IV, Pydantic everywhere).

### Step 6 — Core: ChatService
**File**: `src/totoro_ai/core/chat/service.py`

```python
class ChatService:
    def __init__(
        self,
        extraction_service: ExtractionService,
        consult_service: ConsultService,
        recall_service: RecallService,
        assistant_service: ChatAssistantService,
        # No ConsultLogRepository here — logging is ConsultService's responsibility
    ) -> None: ...

    async def run(self, request: ChatRequest) -> ChatResponse:
        # 1. classify intent
        # 2. if clarification_needed → return ChatResponse(type="clarification", ...)
        # 3. dispatch by intent
        # 4. wrap result in ChatResponse
        #    - assistant returns str; wrap as message=result, data=None
        #    - others return Pydantic models; wrap as data=result.model_dump()
        # 5. on any exception → return ChatResponse(type="error", ...)
```

`ConsultService.consult()` is responsible for saving the consult log before returning its result — logging lives at the service that owns the consult outcome, not the dispatcher.

**Key dispatch signatures**:
- `ExtractionService.run(request.message, request.user_id) -> ExtractPlaceResponse`
- `ConsultService.consult(request.user_id, request.message, request.location) -> ConsultResponse`
- `RecallService.run(request.message, request.user_id) -> RecallResponse`
- `ChatAssistantService.run(request.message, request.user_id) -> str`

### Step 7 — Route: POST /v1/chat

### Step 11 — ADRs (required before merge)
**File**: `docs/decisions.md`

Add two new ADR entries at the top:

**ADR-052**: Consolidate routes into routes/chat.py, superseding ADR-018  
- Context: Feature 017 introduces a unified /v1/chat entry point. All intent-specific routing moves to ChatService. Individual route modules (extract_place.py, consult.py, recall.py, chat_assistant.py) are deleted.  
- Decision: routes/chat.py is the single route module for all conversational API traffic. ADR-018 is superseded.

**ADR-053**: This repo owns consult_logs table for AI recommendation history  
- Context: Feature 017 needs to persist AI-generated recommendation history for feedback loops and taste model improvement. The product table `recommendations` is owned by NestJS (per Constitution Section VI). Naming the new table `recommendations` would create a write-ownership conflict.  
- Decision: This repo adds `consult_logs` (not `recommendations`) via Alembic. NestJS has no recommendations table — it only owns users and user_settings via TypeORM.
